# Z_DIAG — Standalone diagnostic tool for FLINT hardware

## Зачем создана

В FLINT 2.77 обнаружен баг: при повторной калибровке (homing) после первого прогона рисования, ось Z не доходит до датчика на ~4354 шага. Первый прогон работает идеально.

Из лога `FLINT32-12.03.2026_1.LOG`:
```
[START] POST-PROC k=1 lastn=0 xcur=22450 ycur=690 zcur=9096 doptr=1
[HOMING] dir=0 Zexpect=8132 Zactual=3778 Zdelta=4354 sens=3F
```

**Важно**: баг есть ТОЛЬКО в перекомпилированных версиях (из `Flint 2.77 src` и `Flint 2.77 (32bit) src`). Оригинальный бинарник из `!GENESIS\Flint 2.77` работает без ошибок.

Z_DIAG — отдельная программа для диагностики ISA-карты и оси Z **без запуска FLINT**. Запускается на целевой машине (Pentium III, DOS), выполняет серию тестов, результаты логирует в `Z_DIAG.LOG`.

---

## Целевая система

- **Компьютер**: Pentium III, 128 МБ RAM
- **ОС**: MS-DOS (загрузка из Windows 98 SE)
- **ISA-карта**: базовый адрес 0x300
  - 3× 8254 таймера (PT0, PT1, PT2)
  - 2× 8255 PPI (PPI, PPI2)
  - RG (регистр управления моторами), LRG (лазер)
  - CMOS-счётчик точек
- **Компилятор**: Borland C++ 4.52 (16-bit, large model)

## Компиляция

### На DOS / Windows 9x (нативно)
```
COMPILE.bat
```
Ищет компилятор в `..\BC45\BIN\` или `C:\BC45\BIN\`. Результат: `ZDIAG\Z_DIAG.EXE`.

### На Windows NT+ (через DOSBox-X)
```
COMPILE.bat
```
Автоматически запускает DOSBox-X (ищет в `..\DOSBox-X\` или в PATH). Результат: `ZDIAG_DDMMYY_HHMMSS\Z_DIAG.EXE` + `RUN.bat`.

### Запуск на целевой машине
Скопировать каталог `ZDIAG_*` на машину, запустить `RUN.bat` (или `Z_DIAG.EXE` напрямую из DOS).

## Авто-диагностика (A)

Запускает все тесты в оптимальном порядке:

| Фаза | Тесты | Моторы? | Описание |
|------|-------|---------|----------|
| 0 | 17, 15 | Нет | Среда, CPU тайминг (работает даже без ISA-карты) |
| 1 | snapshot, 1, 3, 6 | Нет | Чтение портов, счётчики, осциллятор |
| 2 | 9 + inline check | Z | Повторяемость хоминга, надёжность датчика |
| 3 | 2, 4, 5, 8 | Z | PB6, целостность, термо, воспроизведение бага |
| 4 | 12, 16, 18 | Нет | Cross-talk, прерывания, стресс-тест I/O |
| 5 | 10, 13, 11 | XYZ | **XYZ integrity, шум датчиков, имитация process()** |
| 6 | 14 | Z | Гравитационный дрейф |

Если датчик Z ненадёжен (spread > 20 в фазе 2), все результаты моторных тестов помечаются как SUSPECT.

## Гипотезы бага и какие тесты их проверяют

| # | Гипотеза | Тест(ы) |
|---|----------|---------|
| 1 | **Прерывание между lo/hi байтами outportb** — INT 08h портит пару записей в 8254 → счётчик получает мусорное значение → потеря шагов | **16, 18** |
| 2 | Cross-talk: K2z и K2x на одном чипе PT0, запись K2x портит K2z | **12** |
| 3 | Шум Z-датчиков при XY-движении → ложное срабатывание end-stop в move() | **13** |
| 4 | Шаги теряются только при одновременном XYZ (тайминг K2 другой) | **10, 11** |
| 5 | delay() откалиброван неправильно → dT коррекция неверна | **15** |
| 6 | Термическая усталость мотора | **5** |
| 7 | Гравитационный дрейф при паузе между прогонами | **14** |
| 8 | Датчик Z нестабилен | **9** |
| 9 | Аппаратная деградация | **4, 6, 7** |

### Главная гипотеза (#1)

Оригинальный бинарник (скомпилирован Borland C++ 4.5x с определёнными опциями) генерирует более плотный машинный код, где пара `outportb(lo); outportb(hi);` не разделяется прерыванием. При перекомпиляции с трассировкой (TRACE/VFY_COUNTERS) или другими опциями оптимизации в коде появляются дополнительные инструкции → окно для INT 08h увеличивается → примерно 1 из 10000 записей в 8254 получает мусор → шаги теряются.

Тест 16 напрямую проверяет: если counter writes с CLI/STI не портятся, а без — портятся, это подтверждает гипотезу.

## Параметры тайминга (по умолчанию)

Взяты из реального лога `FLINT32-12.03.2026_1.LOG`, строка `[PROC] CFG`:

| Параметр | Значение | Откуда |
|----------|----------|--------|
| `l_period` | 35 мс | Период лазера |
| `dT` | 6 мс | Коррекция CPU timing |
| `step_freq` | 1000 Гц | Макс. частота шагов |
| `f_const` | 0 | Переменная скорость (не const) |

Эти значения используются в `calc_k2()` и `seg_time()` для тестов 10 и 11 (XYZ-движение). Можно изменить через меню `G` перед запуском тестов.

## Безопасность

- Лазер **ВЫКЛЮЧЕН** (LRG=0, PC5=0, PC6=1)
- Все моторные тесты начинаются с хоминга к датчикам
- **ESC** — аварийная остановка в любом тесте
- Расстояния в пределах нормальных рабочих: Z≤8580 шагов (макс. Zscale=16384)
- Концевые датчики проверяются во время каждого движения
- Без ISA-карты программа работает (WARNING в логе, не блокирует)

## Интерпретация результатов

### Тест 16/18 показывает corruption без CLI:
→ **Подтверждена гипотеза #1**. Нужно обернуть все пары `outportb` в `move()` / PIC.CPP через `disable()`/`enable()`.

### Тест 12 показывает cross-talk:
→ Аппаратная проблема или неисправный 8254. Замена чипа.

### Тест 13 показывает Z-триггеры при XY:
→ Шум на линиях датчиков. Проверить экранирование кабелей.

### Тест 10/11 показывает потерю шагов, а тест 4 — нет:
→ Проблема в XYZ-взаимодействии (тайминг или электрика).

### Тест 15 показывает delay() неточность:
→ Перекомпиляция с TSR или другим DOS-конфигом меняет delay() calibration. Пересчитать dT.

### Тест 14 показывает drift:
→ Гравитация + отключение шаговиков между прогонами. Не отключать питание моторов.

### Все тесты OK:
→ Проблема может быть в тайминге, который слишком редок для 200/1000 итераций. Повторить тесты 16/18 с большим числом итераций или под нагрузкой.
